# 台北網球公開 MVP 設計規格

日期：2026-07-17
狀態：設計已於對話確認；待創辦人審閱本文件後，才進入實作計畫
範圍：公開 Web MVP；首發只服務台北地區的網球揪局

## 1. 一句話

做一個讓人打開後立刻看見「台北哪裡、何時、缺幾人可以打網球」，並能安全地申請加入、由主揪核准後交換 LINE 聯絡方式的公開 Web 服務。

## 2. 已確認的決策

| 主題 | 決策 |
|---|---|
| 首發市場 | 台北地區、網球；不在第一版同時上其他運動或其他城市 |
| 上線方式 | 直接公開上線，不設封閉 beta 或等候名單 |
| 分發 | 由創辦人經 LINE 網球群組、Facebook 網球社團分享；先遵守各群規則並取得管理員同意 |
| 核心價值 | 使用者能快速看到可加入的球局，而非只瀏覽永久公開的球友名單 |
| 核心互動 | 看球局 → 點選加入 → 主揪核准／婉拒 → 核准後雙方交換 LINE |
| 聯絡安全 | LINE ID 是後端隱私資料；未獲雙方同意前，任何匿名或已登入使用者都不可讀取 |
| 未來擴展 | 資料邊界預留 sport 概念；真正新增第二種運動時才設計通用場地模型 |
| 生態研究 | 不阻擋公開上線；以公開資料、短表單與少量非同步文字交流持續補強 |

## 3. 問題與目標使用者

### 3.1 要解的問題

台北網球的揪球資訊散落在 LINE、Facebook、熟人網絡與各球場的訂場制度裡。想打球的人難以快速知道近期有哪些真實球局；手上有場地與時間的主揪，則可能缺 1–3 位合適的人。

產品不試圖取代訂場系統、俱樂部或教練課程，而是補上「公開發現可加入球局」與「雙方同意後聯絡」這一段。

### 3.2 首批使用者

- **找球局者：** 從 LINE／Facebook 社群點進來，想在台北近期打一場網球的人。
- **主揪：** 已經有球場或可處理訂場、知道時間，缺少 1–3 位球友的人。
- **群組管理者／固定團主揪：** 不是第一版後台角色，但可能是分發與信任的關鍵合作對象。

### 3.3 成功定義

公開上線後的前兩週，以以下漏斗衡量，而不是下載量或純註冊數：

`看見球局 → 點選加入 → 主揪核准 → 回報成局`

建議的初步觀察門檻：20 位真實使用者、8 個真實球局、3 次核准加入、至少 1 次回報成局。這些是判讀產品方向的 KPI，不是延後上線的門檻。

## 4. 產品定位與範圍

### 4.1 產品定位

> 台北網球的即時揪球看板。

地圖是發現工具，球局才是產品的原子單位。公開球友檔案可在後續保留作輔助探索，但不應取代具體的日期、時間、球場與缺額資訊。

### 4.2 第一版必做

- 匿名瀏覽公開的近期球局。
- 以地圖與列表探索；可依區域／球場、時間、程度與打法篩選。
- 登入後建立球局，填寫球場、開始時間、打法、程度範圍、缺額與備註。
- 登入且完成最小個人檔案後，對球局送出加入申請。
- 主揪可接受或婉拒申請；滿額、取消與過期都有清楚狀態。
- 核准後雙方看見彼此的 LINE ID，並可複製開場訊息；不建立站內聊天室。
- 「我的球局」頁面，分開顯示自己開的、已申請的與已核准的球局。
- 成局後讓參加者回報「打成了」，作為最小可用的結果訊號。
- 基本檢舉、聯絡入口、隱私說明與錯誤恢復畫面。

### 4.3 第一版不做

- 訂場、付款、退款、抽成或場地庫存同步。
- 站內即時聊天、推播通知、演算法配對、評分／積分或教練媒合。
- 自動擷取、重貼或儲存 Facebook 私密社團、LINE 群組的貼文或個資。
- 台北以外城市，或第二種運動的正式使用者介面與營運。

## 5. 使用者體驗與狀態

### 5.1 找球局者

1. 從社群連結開啟網站，匿名即可查看近期球局。
2. 以地圖或列表找到合適的球局，看到球場、開始時間、打法、程度、缺額與備註。
3. 點選「加入」；若尚未登入或檔案不完整，完成後必須回到原球局與原意圖。
4. 送出申請後看到「等待主揪回覆」，可在「我的球局」主動查看狀態或撤回。
5. 主揪核准後，才顯示雙方 LINE ID 與開場訊息。
6. 球局結束後，可回報是否打成。

### 5.2 主揪

1. 開球局時填結構化資料，而非只有自由文字：球場、開始時間、打法、程度範圍、缺額（1–3）與備註。
2. 在「我的球局」看到申請者的最小資訊：暱稱、NTRP、打法與常打球場；尚未核准前不顯示對方 LINE。
3. 接受或婉拒申請；接受後雙方互相可見 LINE。
4. 可取消球局；系統在開始時間後將其過期或導向成局回報。

### 5.3 狀態機

```text
session.status
open ──(缺額填滿)──> full
  │                    │
  │                    └──(已核准者在開始前退出)──> open
  ├──(主揪在開始前取消)──────────────────────────> cancelled
  └──(主揪在開始後回報打成)──────────────────────> played

open 或 full ──(開始後 24 小時仍未回報打成)──> expired

participant.status
requested ──(主揪接受)──> accepted
    │                       │
    ├──(主揪婉拒)──> declined
    └──(本人撤回)──> withdrawn
accepted ──(本人退出)──> withdrawn
```

只有主揪可在開始後把球局標記為 `played`；已核准的參加者可另記自己的到場確認，供後續衡量，但不單獨改變球局狀態。已填滿的球局不可再接受新申請；若已核准者在開始前退出，球局回到 `open`。若未來要支援候補，需新增明確的 waitlist 狀態，不能在第一版隱性超賣。

## 6. 資料、權限與隱私設計

### 6.1 建議的資料邊界

| 實體 | 責任 |
|---|---|
| `profiles` | 使用者最小公開資料與私密 LINE ID；擁有者可編輯 |
| `sports` | 先只種 `tennis`；作為未來擴展邊界 |
| `courts` | 現有網球場目錄與座標；首發不用提早泛化成所有運動場地 |
| `sessions` | 一個具體球局：`sport_id`、court、時間、打法、程度範圍、缺額、備註與生命週期 |
| `session_participants` | 主揪與申請者對球局的角色、狀態與成局回報 |
| `session_discovery` | 專供匿名探索的最小公開 view，不含聯絡資訊與不必要的 profile 識別資料 |
| `session_contacts` | 僅在主揪與該位申請者同一球局、雙方均為 accepted 時，提供彼此 LINE 的受限 view |
| `reports` | 對球局或使用者的檢舉紀錄；第一版由人工處理 |

### 6.2 權限規則

- 匿名者只能讀近期、公開、未過期的 `session_discovery`，不能直接讀 profiles、participants、contacts 或原始 session 資料表。
- 登入者只能修改自己的檔案、送出／撤回自己的申請，或管理自己主揪的球局。
- 只有主揪能把 `requested` 改為 `accepted` 或 `declined`。
- `session_contacts` 必須在資料庫層確認閱覽者與對方是同一球局的主揪／已核准申請者配對；不能只由前端按鈕決定是否顯示。第一版中，已核准的兩位客人不會因同場就互相看見 LINE。
- 建局、申請、接受、取消等跨表操作要採資料庫函式／RPC 或同等原子化方案，避免半成功資料與滿額競態。

### 6.3 現有原型的必要轉換

目前的 `partner_requests` 與「快速約球」是單向貼文／直接交出聯絡方式的模型。公開 MVP 必須以 `sessions` 與參與者狀態取代它，並將 `line_id` 從匿名可讀的 discovery 資料中移除。這是公開上線的 P0 安全門檻。

## 7. 技術架構與品質目標

### 7.1 架構方向

保留既有 Vite + vanilla JavaScript + Supabase 架構，不為這一版重寫框架。重整責任邊界即可：

- UI 模組：探索、開球局、我的球局、個人檔案。
- `dataApi`：唯一資料邊界；UI 不直接呼叫 Supabase table 或 view。
- Supabase：Auth、Postgres、RLS、受限 view 與原子化 RPC。
- Playwright：端對端使用流程；pgTAP：資料庫權限與狀態轉移。

### 7.2 性能與可靠性預算

- 一般手機網路下，首次頁面在 3 秒內可開始操作。
- 探索查詢只讀目前地圖範圍與近期時間窗，不一次載入所有歷史球局。
- 公開探索只取顯示所需欄位，並為 `status + start_at`、`court_id + start_at` 等查詢建立索引。
- 篩選、地圖／列表切換需立即有回饋；網路更新時顯示 loading、empty、error 與 retry。
- 加入時若球局剛滿額、取消或過期，後端以最新狀態拒絕並回傳可理解訊息；前端重新整理該球局。
- OAuth／補完檔案後保留原本的「加入哪一局」意圖，避免使用者在登入流程中流失。

### 7.3 測試門檻

- pgTAP：匿名與未核准使用者無法取得 LINE；狀態轉移與滿額規則不可繞過。
- Playwright：匿名瀏覽、登入回跳、開局、申請、接受、取消、聯絡揭露與成局回報的完整雙人旅程。
- 前端：篩選／空狀態／錯誤重試、手機版流程與地圖金鑰失效 fallback。
- 發布前：production build、E2E、資料庫測試、資料庫 migration 與 RLS 的 hosted QA。

## 8. 台北網球生態研究（與公開上線並行）

這不是全台百科計畫，也不應卡住產品上線。目標是在實際上線與分發時，選出最容易形成流動性的球場與社群。

### 8.1 低摩擦做法

- 使用官方公開資料建立 15 座左右重點球場卡：訂場方式、開放時段、費用、已知固定活動、查證日期與來源。
- 只在可行時，透過文字和 1–3 位熟人、主揪或群組管理員確認痛點；不要求正式訪談。
- 經群組管理員同意，以很短的表單或公開頁收集「有場缺人」與「想加入」的自願訊號。
- 不爬群組貼文、不儲存未同意者的身份資料，也不將未驗證的「俱樂部佔場」作為公開標籤。

### 8.2 研究產出

每座球場的生態卡至少包含：區域／交通、訂場模式、公開可加入性、可能的固定團或課程、資訊來源、查證日期、是否適合作為試點。研究應最終回答：先服務哪 2–3 座球場與透過哪 2 個社群分發。

## 9. 公開上線與營運

### 9.1 上線方式

公開 MVP 從第一天就允許瀏覽、註冊、開局與加入，不使用「封閉 beta」作為對外名稱。分發仍採有節奏的方式：先將分享集中在台北網球的相關群組，讓創辦人能及時處理問題、回覆檢舉並觀察真實漏斗。

### 9.2 最低上線門檻

- 核准前不得洩露 LINE 或其他聯絡資訊。
- 開局、申請、接受／婉拒、撤回、取消與過期流程完整可用。
- 有基本隱私說明、聯絡／檢舉入口、空狀態與錯誤復原。
- 僅顯示真實且經主揪同意公開的球局；不得用假球局製造熱度。

### 9.3 觀察指標

| 漏斗 | 指標 |
|---|---|
| 供給 | 每週公開球局數、主揪數、球局取消率 |
| 需求 | 每週加入申請數、申請者數、申請後撤回率 |
| 媒合 | 核准率、從申請到核准的時間 |
| 結果 | 回報成局數、30 天同伴回訪率（後續） |
| 品質 | 檢舉率、載入／操作錯誤率、首次可操作時間 |

每週只挑漏斗中最大的單一摩擦點改善一次，避免同時擴區域、加運動種類與堆新功能，導致無法判斷哪個改動有效。

## 10. 未來擴展原則

「不受限於任何運動」是長期架構方向，不是首發市場。現在用 `sport_id`、通用的 session／participant 概念保留選項；當第二種運動真的具備獨立的社群、場地資料與揪局流動性時，再建立 `venues`／`playing_spaces` 等通用場地模型與運動專屬欄位。

擴運動的觸發條件應是台北網球的核心漏斗已可反覆成局，而不是單純因為資料模型能容納第二種運動。

## 11. 實作順序（供後續計畫展開）

1. 將本設計轉為逐檔案、逐測試的實作計畫。
2. P0：session schema、RLS、contact secrecy 與原子化狀態轉移。
3. P1：探索、開球局、加入、主揪審核與我的球局 UI。
4. P1：雙人 E2E、資料庫權限測試、效能／錯誤恢復與 hosted QA。
5. 公開上線、群組分發、漏斗觀察與低摩擦球場生態研究。
6. 依數據選擇後續項目：球場深度指南、通知、候補、教練合作或第二種運動。

## 12. 風險與明確取捨

- **冷啟動：** 地圖可能一開始很空；誠實呈現空狀態並協助主揪發真實球局，不用假資料掩蓋。
- **個資與騷擾：** 聯絡資訊只在雙方核准後由資料庫層放行；保留檢舉入口，第一版人工處理。
- **場地規則差異：** 產品只記錄經查證的公開資訊，不承諾訂場成功，也不替使用者處理付款或場地糾紛。
- **框架／泛化誘惑：** 不為假設中的多運動重寫前端或大改資料庫；先確保一條網球揪局迴圈可以跑通。
- **通知缺口：** 第一版可接受主揪透過「我的球局」主動查看；若核准等待時間是主要流失點，再評估通知層。
